Oracle体系结构——002

引言

初学数据库的时候最害怕不知道改了什么东西导致数据库起不来了，这种事情在我学习的过程中发生了无数次，这里我将简单的介绍oracle的启动过程

Oracle参数文件

在数据库启动时会在一个特定配置文件中指定Oracle实例的初始化参数，这个文件即是参数文件。每次启动数据库的时候要求参数文件是完好的，参数文件中的参数会用于SGA对内存设置的自动调整

参数文件分为静态参数文件和动态参数文件，多放在$ORACLE_HOME/dbs下

我们现在大多都使用动态参数文件，（在9i之前使用静态参数文件）；而静态参数文件需要重启实例才能生效

动态参数文件，也称为服务器参数文件（server parameter file）或SPFILE。它是一个包含初始化参数的二进制文件。在SPFILE中可说明实例的内存限制、控制文件位置，归档日志是否保存及保存的位置，以及其他一些决定Oracle数据库服务器行为的设置，他们多以spfile+SID.ora命名，如上图的 spfileorcl.ora。动态参数文件的优点就是可以在实例启动的情况下有一部分参数可以直接修改并生效

在9i之前我们使用静态参数文件，也被称为初始化参数文件（PFILE），是一个包含所有初始化参数列表的文本文件，可以用文本编译器直接修改。他们多命名为init+SID.ora，修改后要重启数据库实例才能生效

SPFILE与PFIE的区别

SPFILE:

  1. 二进制文件，不可以用编辑器修改
  2. 在linux下可使用strings命令查看
  3. 命名方式：spfile+SID.ora
  4. 有一部分参数可以直接修改并参数
  5. 9i版本之后支持

PFILE：

  1. 文本文件，可以直接用编译器修改
  2. 修改参数必须关闭实例，下次重启才能生效
  3. 命名方式：init+SID.ora
  4. 多在9i之前使用

如何查看数据库以哪种参数文件启动

查看参数spfile，如果这个参数有值，那么就是动态参数文件启动，反之为PFILE启动

PFILE和SPFILE是可以互相生成的

首先请查看Oracle关于CREATE PFILE的联机文档

在11g的版本中，可以通过内存生成spfile和pfile


当然，通过spfile启动的实例也会出现相同的问题

startup时读取参数文件，找到$ORACLE_HOME/dvs目录，顺序是优先spfile启动，没有spfile才从pfile启动。当有了spfile时，pfile一般留作备用，特殊情况也可以使用pfile启动

动态参数与静态参数

Oracle11g有300多个常用参数；当spfile读到内存后，有一部分参数是可以直接在内存中修改，并立即生效，这样的参数又叫动态参数。除了动态参数都是静态参数，静态参数的修改必须修改init.ora文件然后重启数据库才能生效

修改spfile的方法


有三种可以修改动态参数值的方法，分别使用ALTER SESSION、ALTER SYSTEM和ALTER SYSTEM …. DEFERRED 命令


1. 使用 ALTER SESSION命令

ALTER SESSION命令允许在发布命令的会话中更改动态参数的值。ALTER SESSION 命令只是用来临时更改参数的值



2. 使用ALTER SYSTEM语句

ALTER SYSTEM语句会更改所有会话的参数值。但是这些更改仅在实例运行中存在。除非修改init.ora文件或使用SPFILE，否则如果重启数据库，更改会失效



第一种： scope=memory 参数修改立即生效，但不更改spfile文件

第二种： scope=spfile 修改了spfile文件，重启后生效（如果不使用此参数，将无法更改静态参数）

第三种： scope=both 前两种都满足

并不是所有参数都可以使用scope=memory的，判断方法在后面介绍。
如果不写scope= 缺省值为both

3. 使用ALTER SYSTEM …. DEFERRED 语句

ALTER SYSTEM …. DEFERRED语句将使一个新值对所有会话生效，但不是立即生效。它仅对执行此语句后启动的会话有效。所有当前打开的会话将继续使用旧的参数值

数据库启动阶段


数据库启动分三个阶段：nomount阶段、mount阶段、open阶段；当我们刚进入系统没有启动实例的时候叫做空闲进程，此时系统中是没有oracle进程的

我们可以查看数据库当前的启动阶段



nomount阶段


在sqlplus会话中使用startup nomount命令启动数据库，这样启动仅有实例运行。如果以这种方式打开数据库将不会读控制文件，而且数据文件不打开。操作系统启动Oralce后台进程，并且给Oracle分配SGA。事实上只有instance本身在运行



mount阶段

在启动过程中，实例启动后的下一步是安装数据库。在安装步骤中，Oracle把实例与数据库关联。Oralce打开并读取控制文件，获取database files和redo log files的名称和位置。在nomount阶段可以使用ALTER DATABASE MOUNT命令进入该阶段

进入mount阶段就可以读取很多视图了，在进行诸如全数据库恢复、更改数据库归档模式或重命名数据库文件这一类活动时，通常需要进入mount阶段修改

open阶段

启动过程的最后一步是打开数据库，用数据文件（datafile）和CTL的头部进行对比，比较scn确定一致性，如果一致就可以打开，否则需要实例恢复。当数据库以open模式启动的时候，所有有效用户都可以连接数据库并进行操作了；在此步骤之前一般用不能连接数据库。进入open阶段代表数据库已经启动完成

数据库的关闭

关闭数据库有四种可用的方式，分别是SHUTDOWN NORMAL、SHUTDOWN TRANSACTIONAL、SHUTDOWN IMMEDIATE、SHUTDOWN ABORT 四个命令

SHUTDOWN NORMAL


在使用SHUTDOWN NORMAL命令关闭数据库时，Oracle将在关闭数据库之前等待所有用户断开与数据库的连接。如果某个用户在登录数据库之后休假了一直挂着，而你却使用了SHUTDOWN NORMAL命令，那么在该用户休假回来关闭自己的连接之前数据库将一直运行。而normal方式也是Oracle数据库的默认关闭方式

- 一旦发布此命令，再不能对数据库进行新的用户连接
- 在关闭数据库之前，Oracle等待所有用户退出Session（会话）
- 重启数据库时不需要实例恢复，因为会把redo buffer和db buffer都写入磁盘
- Oracle关闭数据库文件并终止后台进程，Oracle的SGA被解除分配


SHUTDOWN TRANSACTIONAL

如果不想为了等待用户断开session而花费时间，可以使用SHUTDOWN TRANSACTIONAL命令。Oracle将在断开所有用户与数据库的连接前等待所有的活动事物完成，然后关闭数据库

  • 一旦发布此命令，不允许任何用户进行新连接
  • 现有用户不能启动新事物，并且将断开连接
  • 如果有一个活动的事物，在断开改用户的连接前，Oracle将等待直到该事物完成
  • 在所有事物完成后，Oralce关闭实例并释放内存，Oracle将所有redo buffer和db buffer写入磁盘
  • 再启动不需要实例恢复

    SHUTDOWN IMMEDIATE

前面的两种并不常用，而SHUTDOWN IMMEDIATE为最常用的关库命令。在SHUTDOWN IMMEDIATE方式下，Oracle将不再等待用户的退出和事物的执行完毕，而是直接回滚所有事物并断开连接，之后关闭数据库

  • 一旦发布此命令，不允许任何用户进行新连接
  • 立即断开所有的用户session
  • 终止所有正在执行的事物
  • 对于所有当前执行的事物，Oralce将进行回滚，使数据库保持一致。这个回滚动作也就是SHUTDOWN IMMEDIATE并不能立即关闭数据库的原因，当然如果没有活动事物，那么SHUTDOWN IMMEDIATE还是可以很快的关闭数据库
  • 再启动不需要实例恢复

SHUTDOWN ABORT

生产环境下决不能使用此命令！SHUTDOWN ABORT既不断开session也不回滚transaction，直接关闭数据库，一般数据库出现严重宕机时才会考虑使用此命令“跳水”

  • 一旦命令发布，不允许进行新连接
  • 现有会话终止，不管是否有活动的事物
  • 不回滚被终止的事物
  • 不将redo buffer和db buffer写入磁盘
  • 终止后台进程、立即释放内存并关闭数据库
  • 在重启时需要实例恢复，在大多情况下不需要显式的recover，因为数据库会自己进行实例恢复（实例恢复在之后的备份还原中详细介绍）

STARTUP FORCE

硬性重启，可以跟参数，相当于SHUTDOWN ABORT后再输入startup [nomount|mount|open]，参数即使数据库启动的三个阶段，不写的话就直接运行到open状态

Oracle体系结构——001

引言

dba最头疼的莫过于数据库的故障与数据迁移，而oracle到目前为止能成为世界占有率最大的数据库也正是因为其数据的可靠性和完备的备份恢复机制，而这些功能是需要一段时间细心学习掌握的，想要更加深入的学习oracle首先要学习他的体系架构，了解其组成才能更好的应用并维护。

为了更好的习惯Oracle，这里的一些特定名词使用英文，这样能更好的帮助我们在维护时理解错误情况并查阅联机文档

Oracle网络架构及应用环境

Oracle有一张非常经典的体系结构图

数据库启动的第一步便是加载实例（Instance），读取参数文件并把一些程序调到内存中。实例由SGA（System Global Area）和后台进程组成，也就是上图的粉色框框中的内容

所以按照顺序，我们第一步就来讲解Instance

Instance

在学习Instance之前必须明白一个概念，即Oracle Instance与Oracle数据库之间的区别，两个术语经常被混淆但事实上他们完全不是一个东西
Oracle数据库由文件组成，即数据文件（datafiles）和oracle的系统文件（system files），但这些文件本身没有什么意义，这就需要一套系统来提供处理能力和资源来操作这些文件中的数据，而这一组特定的、由Oracle在服务器上创建的进程与操作系统分配给它的存储想结合时，就得到了Oracle Instance。

而我们常说的Oracle启动了，其实就是再说Oracle实例启动了，我们通过实例来访问数据库而一个数据库对应多个实例（常见于Oralce RAC），但一个实例只能对应一个数据库

Instance由SGA和后台进程（backgroud process）组成，首先介绍SGA

系统全局区（System Global Area）

以下简称SGA

首先请查看Oracle关于SGA的联机文档

5.1.4.1 System Global AreaOracle联机文档

其中一段话为：

The System Global Area (SGA) is a group of shared memory structures, known as SGA components, that contain data and control information for one Oracle Database instance. The SGA is shared by all server and background processes. Examples of data stored in the SGA include cached data blocks and shared SQL areas.

SGA-Oracle联机文档

机翻一下就是：

系统全局区域 (SGA) 是一组共享内存结构,称为 SGA 组件,其中包含一个 Oracle 数据库实例的数据和控制信息。SGA 由所有服务器和后台进程共享。存储在 SGA 中的数据示例包括缓存的数据块和共享 SQL 区域。

我们可以将其粗略的理解为Instance中各个器官的集合，SGA中的主要器官共有6个

1. Shared Pool 共享池

Shared Pool是对SQL、PL/SQL进行语法解析、编译、执行的内存区域，它由库缓存（library cache）和数据字典缓存（data dictionary cache）组成，某种程度上讲Shared Pool的大小直接影响Oracle的性能。

在library cache 中保存了SQL、PL/SQL语句的执行计划，当使用一样的语句的时候就不用再解析了（硬解析），也就减少了sql的成本

数据字典（data dictionary）是Oracle非常重要的部分，由Oracle维护，因为需要经常访问的缘故ORacle将其放到缓存中，即data dictionary cache

虽然Shared Pool能优化Oracle性能，但事实上并不是越大越好，这点在后文中将会讲解，一些工程师会将其设置在500M左右

2. Database buffer cache

Oracle中非常有名的Database buffer cache，以下简称为DB buffer。

Before data stored in the database can be queried or modified, it must be read from a disk and stored in the buffer cache. All user processes connected to the database share access to the buffer cache. For optimal performance, the buffer cache should be large enough to avoid frequent disk I/O operations.

SGAOracle联机文档

在查询或修改数据库中存储的数据之前,必须从磁盘读取数据并存储在缓冲区缓存中。连接到数据库的所有用户进程共享对缓冲区缓存的访问。为获得最佳性能,缓冲区缓存应足够大,以避免频繁的磁盘 I/O 操作。

数据库操作的时候会将数据文件（datafile）的数据块读到db buffer中，在dbbuffer中生成一个副本（读进程server process），如果这个数据块有改动就称这个块为脏块，在之后某段时间将他读回datafile进行更改（写进程DBWR）。而且当他人访问时如果该数据块整好在内存中，就可以直接读取

db buffer中的有几个重要参数：

1. db_cache_size

指定标准块大小的内存空间

2. db_nk_cache_size

指定非标准块大小内存空间

这里的nk就是指上面非标准块的名称，n代表数据块的大小
例如我们要定义一个16k大小的块参数，那么块就会对应命名为 db_16k_cache_size

3. db_keep_cache_size

首先请查看Oracle关于db_keep_cache_size的联机文档

1.79 DB_KEEP_CACHE_SIZEOracle联机文档

在oracle中使用了一种算法（LRU Least Recently Used）将db buffer中长时间不用的块踢出去，而我们有一些表即使不常用也不想被清除处db buffer，如系统使用的一些字典表等，而db_keep_cache_size就存放了这些表并使他们不会被清除出db buffer



4. db_recycle_cache_size

首先请查看Oracle关于db_recycle_cache_size的联机文档

1.85 DB_RECYCLE_CACHE_SIZEOracle联机文档

它与db_keep_cache_size相反，用于存放一些我们不希望它存放在db buffer中的表，这些表通常很大且不常访问，将其放在db buffer会影响性能

3. Redo log buffer

(Redo log将在之后介绍，感兴趣可先查看oracle联机文档加以了解，其中scn、ARCHIVELOG、LGWR 在之后都会有所讲解)

首先请查看Oracle关于 redo log的联机文档

What Is the Redo Log?Oracle联机文档

日志条目（redo entries）记录了数据库的所有修改信息（包括DML和DDL），为的是数据库恢复，日志条目首先产生于日志缓冲区（Redo log buffer）。日志缓冲区一般较小，它是以字节为单位的，但是它极其重要

Redo log buffer大小最好为3M（原因会在LGWR处介绍）。需要手动设置，设置之后需要重启数据库，显示时按照字节显示

使用 SHOW PARAMETER log_buffer;查看Redo log buffer当前大小

Redo log buffer有一个最小值，我们可以用log_buffer=1的方法设置



之后重启即可
在实验中我们可以使用startup force重启，但是生产环境中千万不要使用该命令！

4. Large pool（可选）

What is the Oracle large pool and how is the large pool used within the SGA?推荐阅读

为了进行大的后台进程操作而分配的内存空间，与shared pool管理不同，主要用于并行查询、RMAN备份恢复、以及共享链接等

虽然是可选的，但是并行查询的时候需要用到，并行查询需要机器有多个cpu，将一个select语句分割开查询，可以大幅度优化查询速度（可达100倍）

SGA的自动管理


Oracle10G开始有了SGA自动管理功能，相关的pool可以动态调整大小
参数：

sga_max_size: SGA的最大物理空间
sga_target:SGA最大可达到空间，sga_target<=sga_max_size,设定这个值>0时就使能了SGA的自动管理

sga_target是sga可使用的物理空间而sga_max_size是sga使用空间的上线，sga_target不可以超过sga_max_size


可以查看参数workarea_size_policy来确定是否开启了自动管理，当value为AUTO则说明开启，数据库是默认开启的




也可以使用 show parameter sga;查看其中sga_target的值，如果为0的话，说明开启了sga自动管理


在这里还要交代一个概念：granules（颗粒） 它是oracle内存的最小单位,而SGA分配空间时，分配的大小大多都是granules 的整数倍数
当sga_max_size < 1G 时 granules为4M

当sga_max_size > 1G 时 granules为16M

我们可以查看动态试图v$sgainfo来查看当前SGA对不同Pool的分配情况





Oracle Process


Oracle的进程分为三种：1）user process;2)server process;3)background process

user process


客户端进程，用户登录时候就会生成一个user process,一般分为三种形式：
1. sqlPlus: 使用sqlplus登录
2. 应用程序：如plsql Developer，还有你的应用项目
3. web方式：OEM访问

有user process造成的会话中止，系统将自动回滚这个会话上的处于活动状态的事务，如没提交的DDL操作

我们使用sqlplus登录时可以使用OS命令来查看用户进程
如linux使用ps -ef|grep sqlplus 来查看

ps是linux中常见的查看进程命令，详情请查看

ps命令linxu命令大全




server process



server process:服务端的进程，user process不能直接访问Orale，必须通过相应的server process访问实例，进而访问数据库

现在大多一个server process对应一个user process ，是一对一关系，当一对多关系时候，叫做共享进程

Plsql Developer登录时就会产生一个server process，新建一个窗口后又会再新建一个server process

我们使用sqlplus登录时可以使用OS命令来查看服务端进程
如linux我们可使用ps -ef|grep LOCAL 来查看


在linux中我们可以判断此链接是本地链接还是远程链接，在这张截图中出现了四行其中（LOCAL=YES）为本地链接，而其余三行（LOCAL=NO）为远程链接

可以使用动态视图v$process来查看Oracle中的进程


其中字段background 为1 说明他是一个后台进程，为null则为server process



还有一种方法就是使用OS的命令查看所有进程，例如linux可使用ps -ef|grep ora

PMON process


Process Monitor process

当客户端断掉时 pmon会查找对应的pga将其收回，清道夫

SMON process


System Monitor process

实例崩溃时，再次重启需要实例恢复，SMON负责监管实例恢复进程(实例恢复是oracle还原很重要的一步，详情会在之后讲解)

DBWR process




DBWR主要将内存中的数据（脏块）写到datafile中，不是脏块不写；
当发生以下几种状况时，dbwr会写
1. ckpt发生（检查点 之后介绍）
2. 脏块太多
3. db_buffer自由空间(free)不够时
4. 每3秒（3秒是一个阀值，每三秒检查以下是否需要写，如果不需要写就不写等三秒后再检查以下）
5. 表空间redo only/offline/backup 等操作时

Dbwr其实是能不行就不写的，实际的时候也是能不写就不写



注意！oracle中的读进程跟写进程不是一个；Commit不会使dbwr有动作

另附：很多地方叫做DBWN（包括官网），这也是正确的，事实上此进程有多个，从DBW0开始排列



LGWR process



日志写，把redo log buffer里的日志记录到redo log files中（从内存写道磁盘中）

Update语句会产生日志，但是不会将日志写入磁盘，只有写commit 的时候，才会将日志写入到磁盘中（后面几个条件达成时，不需要commit也会写）
LGWR执行的条件：
1. commit
2. 三分之一满（当redo log buffer被占用达三分之一时）
3. 1M满（当redo log buffer被占用达1M时）
4. 先于DBWR写而写（先记后写，即dbwn正好要写入前，执行DBWR写入日志）
5. 3秒（由条件4触发）

之前在redo log buffer部分说过Redo log buffer大小最好为3M，这是因为LGWR每满1M写入一次，而条件2为每三分之一满的时候写入一次，那么当Redo log buffer大小为3M时条件2与3就合并了

3秒是因为dbwr每3s写一下，由于先记后写机制（先再redo log files里记录日志，然后再再data files里写入脏块），而DBWR的写入机制就时每3S写入一次

When a user issues a COMMIT statement, LGWR puts a commit record in the redo log buffer and writes it to disk immediately, along with the transaction’s redo entries. The corresponding changes to data blocks are deferred until it is more efficient to write them. This is called a fast commit mechanism. The atomic write of the redo entry containing the transaction’s commit record is the single event that determines the transaction has committed. Oracle returns a success code to the committing transaction, although the data buffers have not yet been written to disk.

LGWROracle联机文档


由条件1可得知，我们在一次进行多条DDl操作时，应频繁commit，以提高性能



CKPT process

检查点，负责协调读一致性，督促dbwr写脏块，当检查点发生时直接影响dbwr，然后由dbwr间接影响lgwr；当数据库正常关机时，会将内存中的所有脏块都写入磁盘，再开机的时候会检查磁盘数据的完整性，此时由ckpt进程来负责检查(check point的详细操作将在之后oracle的备份恢复中详细介绍)

Others


一些其他的后台进程，目前了解即可

  1. arcn：归档模式下，发生日志切换时，把当前日志组中的内容写入归档日志，作为备份历史日志（只要lgwr（负责联机日志写）和arcn（负责联机日志读）操作日志，其他进程与日志文件不接触）
  2. MMON：Oracle自我检视和自我调整的支持进程，与AWR有关（11g new）
  3. MMNL：MMON的辅助进程，与ASN有关（11g new）
  4. MMAN：内存自动管理，在11G中的到加强，负责Oracle内存结构（SGA+PGA）的自动调整
  5. CJQN：与job队列有关

PGA


User process给server process 发出请求，然后在pga中给其分配一块私有空间，PGA和SGA的内存占比是自动调整的
与SGA不同，PGA时独立的，非共享
PGA不属于实例，SGA和PGA组成oracle的内存结构
